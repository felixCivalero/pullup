Approach Options: Landing Page Editor Integration

How to embed the real CreateEventPage into the landing page with expand animation

A

Component Extraction + Portal

How: Extract the CreateEventPage internals into a reusable <EventEditor /> component. On the landing page, render it inside the showcase section at compact size. On expand, use a React Portal to move it into a fixed fullscreen overlay with a smooth CSS animation (transform + scale from the original position).

Pros

  • Clean separation — editor component is reusable
  • Smooth animation with FLIP technique (First-Last-Invert-Play)
  • No route change — stays on landing page, data in memory
  • Overlay can be dismissed to return to landing page

Cons

  • Significant refactor of CreateEventPage (~4500 lines) into extractable component
  • Must handle all the same state, refs, side effects in new context
  • Portal rendering can be tricky with scroll/focus management

Effort: High — major refactor of CreateEventPage

B

Iframe + Animation Shell (Recommended)

How: Keep CreateEventPage at /create exactly as-is but make it publicly accessible (auth guard only on publish). On the landing page, show the DemoEditor mockup in the showcase section. When user interacts, animate the showcase container to fullscreen, then load an iframe of /create (or navigate). Communication between landing page and editor via postMessage or shared localStorage.

Pros

  • Zero refactor of CreateEventPage — it stays untouched
  • Animation is purely on the shell/container — simple CSS
  • Editor gets full viewport naturally
  • Draft localStorage already works for data persistence

Cons

  • iframe feels slightly less native (scrolling, focus quirks)
  • Communication between landing page and iframe is indirect
  • Double memory footprint while both pages are loaded

Effort: Low-Medium — mostly animation + auth changes

C

Route-Based with Animated Transition

How: Make /create publicly accessible (auth only on publish). On the landing page, clicking the editor CTA captures the element position, stores it, then navigates to /create. The create page reads the stored position and plays an "expand from origin" animation on mount. Data persists via localStorage drafts. Publish triggers auth modal → profile step → auto-publish.

Pros

  • Minimal changes to CreateEventPage — just remove auth guard, add publish-time auth
  • Clean architecture — each page owns its route
  • No portal/iframe complexity
  • Existing draft system handles data persistence naturally
  • Back button returns to landing page

Cons

  • Page transition is a navigation (brief flash possible)
  • Animation across route change requires coordination
  • Not truly "inline" — it's a page change with a nice animation

Effort: Low — auth changes + entry animation + auth modal on publish

Recommendation: Option C

Option C gives the best bang-for-effort ratio. The CreateEventPage barely needs changes — we just make /create publicly accessible, move the auth check from page-load to publish-time, and add the auth+profile modal flow. The expand animation from the landing page mockup to the full editor is achievable with a shared-element-style transition. The existing localStorage draft system already handles persistence across the OAuth redirect.

Option A is technically superior but requires a massive refactor of a 4500-line file. Option B works but iframes introduce UX friction. Option C keeps the architecture clean and the diff small.
